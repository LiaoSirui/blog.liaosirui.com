## kube-apiserver 优化

### 通用

kube-apiserver 推荐优化的参数如下：

1. `--watch-cache=true`
2. `--default-watch-cache-size`：默认值 100；用于 List-Watch 的缓存池；建议 1000 或更多；
3. `--watch-cache-sizes`: 系统根据环境启发式的设定；用于 pods/nodes/endpoints 等核心资源，其他资源参考 default-watch-cache-size 的设定； K8s v1.19 开始，该参数为动态设定，建议使用该版本。
4. `--delete-collection-workers`：默认值 1；用于提升 namesapce 清理速度，有利于多租户场景；建议 10；
5. `--event-ttl`: 默认值 1h0m0s；用于控制保留 events 的时长；集群 events 较多时建议 30m，以避免 etcd 增长过快；
6. `--max-mutating-requests-inflight`: 默认值 200；用于 write 请求的访问频率限制；建议 800 或更高；
7. `--max-requests-inflight`: 默认值 400；用于 read 请求的访问频率限制；建议 1600 或更高；

kube-apiserver 以下两个参数可以控制连接数:

```bash
--max-mutating-requests-inflight int           The maximum number of mutating requests in flight at a given time. When the server exceeds this, it rejects requests. Zero for no limit. (default 200)
--max-requests-inflight int                    The maximum number of non-mutating requests in flight at a given time. When the server exceeds this, it rejects requests. Zero for no limit. (default 400)
```

节点数量在 1000 - 3000 之间时，推荐：

```bash
--max-requests-inflight=1500
--max-mutating-requests-inflight=500
```

节点数量大于 3000 时，推荐：

```bash
--max-requests-inflight=3000
--max-mutating-requests-inflight=1000
```

`--watch-cache-sizes`：调大 resources 的 watch size，默认为 100，当集群中 node 以及 pod 数量非常多时可以稍微调大，比如： `--watch-cache-sizes=pods#1000,nodes#1000,endpoints#1000,endpointslices#1000,events#200`

大集群把 `pods/nodes/endpointslices` 调大到 1000~5000。

### 其他

独给 `events` 指定 etcd：`--etcd-servers-overrides=/events#https://etcd-events:2379`，避免事件风暴拖垮主 etcd。

- 审计必开但要分级：`Metadata` 级别为默认，敏感资源（secret/token）用 `RequestResponse`，高频资源（events、leases、endpointslices）直接 `None`。

- `--audit-log-maxsize=100 --audit-log-maxbackup=10 --audit-log-maxage=7`。

- 大集群推荐 webhook 异步模式：`--audit-webhook-mode=batch --audit-webhook-batch-max-size=400`。

- `--v=2`（不要上 4 以上，日志量指数级增长）。

## kube-controller-manager 优化

与 kube-apiserver 通信的 qps 限制，推荐为：

- 调大 `--kube-api-qps` 值：可以调整至 100，默认值为 20
- 调大 `--kube-api-burst` 值：可以调整至 200，默认值为 30

```bash
--kube-api-qps=100          # 默认 20，大集群 200~400
--kube-api-burst=200        # 默认 30，大集群 400~800
```

Controller 并发

```bash
--concurrent-deployment-syncs=50
--concurrent-replicaset-syncs=50
--concurrent-statefulset-syncs=20
--concurrent-job-syncs=20
--concurrent-cronjob-syncs=20
--concurrent-service-syncs=10
--concurrent-endpoint-syncs=10
--concurrent-service-endpoint-syncs=10   # EndpointSlice
--concurrent-resource-quota-syncs=10
--concurrent-namespace-syncs=10
--concurrent-gc-syncs=50
--concurrent-ttl-after-finished-syncs=10
--concurrent-horizontal-pod-autoscaler-syncs=10
```

原则：CPU 核足够就放大 `*-syncs`，Pod/Job 多的把 `job`、`deployment`、`replicaset`、`gc` 调大

其他

- 禁用不需要的 controller：默认启动为`--controllers`，即启动所有 controller，可以禁用不需要的 controller
- 调整 controller 同步资源的周期：避免过多的资源同步导致集群资源的消耗，所有带有 `--concurrent` 前缀的参数

## kube-scheduler 优化

调大`--kube-api-qps` 值：可以调整至 100，默认值为 50

```bash
--kube-api-qps=100
--kube-api-burst=200
```

建议调整

```yaml
apiVersion: kubescheduler.config.k8s.io/v1
kind: KubeSchedulerConfiguration
leaderElection:
  leaderElect: true
  leaseDuration: 15s
  renewDeadline: 10s
  retryPeriod: 2s
clientConnection:
  qps: 200
  burst: 400
percentageOfNodesToScore: 0                 # 0=自动（默认公式），大集群可设 5~10 加速
parallelism: 32                             # 并发打分 goroutine，默认 16
profiles:
- schedulerName: default-scheduler
  pluginConfig:
  - name: NodeResourcesFit
    args:
      scoringStrategy:
        type: LeastAllocated                # 或 MostAllocated（bin-packing）
        resources:
        - name: cpu
          weight: 1
        - name: memory
          weight: 1
  - name: PodTopologySpread
    args:
      defaultingType: System
  plugins:
    score:
      disabled:                             # 关闭用不到的打分插件，降低延迟
      - name: ImageLocality                 # 镜像预热集群可关
    preScore:
      disabled:
      - name: ImageLocality

```

多调度器支持：kubernetes 也支持在集群中运行多个调度器调度不同作业，例如可以在 pod 的 `spec.schedulerName` 指定对应的调度器，也可以在 job 的 `.spec.template.spec.schedulerName` 指定调度器。

## Kubelet 优化

- 设置 `--image-pull-progress-deadline=30m`
- Kubelet 单节点允许运行的最大 Pod 数：`--max-pods=110`（默认是 110，可以根据实际需要设置）

## 压测

压测验证：`kubemark` 或 `clusterloader2` 模拟你的规模，跑 `load` / `density` 测试。
