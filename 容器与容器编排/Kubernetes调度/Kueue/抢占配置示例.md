# Kueue 抢占（Preemption）示例

本例演示 Kueue 的两种抢占场景：

1. Cohort 内抢占（reclaimWithinCohort）：team-b 借走了 team-a 的配额，当 team-a 有新任务时，可以把借出的部分「要回来」，抢占 team-b 中借用配额的 Workload。
2. 同队列内基于优先级抢占（withinClusterQueue）：同一个 ClusterQueue 内，高优先级 Workload 可以抢占低优先级 Workload。

## 拓扑结构

```yaml
Cohort(team-ab)
├── team-a-cq（保底 4 核，最多借到 6 核，开启抢占）
│     └── team-a-queue
└── team-b-cq（保底 2 核，最多借到 5 核，开启抢占）
      └── team-b-queue
```

## 节点准备

给节点打 label 和 taint（和 QUOTA.md 一样）：

```bash
kubectl label node kube-wk-node node-type=cpu
kubectl taint node kube-wk-node dedicated=team:NoSchedule
```

## 创建 ResourceFlavor

```yaml
---
apiVersion: kueue.x-k8s.io/v1beta2
kind: ResourceFlavor
metadata:
  name: cpu-flavor
spec:
  nodeLabels:
    node-type: cpu
  tolerations:
    - key: dedicated
      operator: Exists
      effect: NoSchedule
```

## 创建 WorkloadPriorityClass

用来做「同队列内基于优先级抢占」。

```yaml
---
apiVersion: kueue.x-k8s.io/v1beta2
kind: WorkloadPriorityClass
metadata:
  name: low-priority
value: 100
description: "低优先级"

---
apiVersion: kueue.x-k8s.io/v1beta2
kind: WorkloadPriorityClass
metadata:
  name: high-priority
value: 1000
description: "高优先级"
```

## 创建 Cohort + ClusterQueue + LocalQueue（启用抢占）

关键点在 `spec.preemption`：

- `reclaimWithinCohort: Any`：允许本队列的 Workload 从 Cohort 内其他队列抢占「借出去的」配额。
  - `Never`：不抢
  - `LowerPriority`：只抢比自己优先级低的
  - `Any`：任何都能抢
- `withinClusterQueue: LowerPriority`：本队列内，高优先级可以抢占低优先级。
  - `Never`：不抢
  - `LowerPriority`：只抢比自己优先级低的
  - `LowerOrNewerEqualPriority`：低优先级或者同优先级但更新（后来）的
- `borrowWithinCohort`：可选，允许在抢占时仍然借用。这里为了让例子清晰，先不开。

```yaml
---
# Cohort
apiVersion: kueue.x-k8s.io/v1beta2
kind: Cohort
metadata:
  name: team-ab

---
# team-a：保底 4 核，最多借到 6 核
apiVersion: kueue.x-k8s.io/v1beta2
kind: ClusterQueue
metadata:
  name: team-a-cq
spec:
  cohortName: team-ab
  namespaceSelector: {}
  preemption:
    reclaimWithinCohort: Any        # 可以把借出去的配额抢回来
    withinClusterQueue: LowerPriority # 队列内可以抢低优先级
  resourceGroups:
    - coveredResources: ["cpu", "pods"]
      flavors:
        - name: cpu-flavor
          resources:
            - name: cpu
              nominalQuota: 4
              borrowingLimit: 2
            - name: pods
              nominalQuota: 5

---
# team-b：保底 2 核，最多借到 5 核
apiVersion: kueue.x-k8s.io/v1beta2
kind: ClusterQueue
metadata:
  name: team-b-cq
spec:
  cohortName: team-ab
  namespaceSelector: {}
  preemption:
    reclaimWithinCohort: Any
    withinClusterQueue: LowerPriority
  resourceGroups:
    - coveredResources: ["cpu", "pods"]
      flavors:
        - name: cpu-flavor
          resources:
            - name: cpu
              nominalQuota: 2
              borrowingLimit: 3
            - name: pods
              nominalQuota: 5

---
apiVersion: kueue.x-k8s.io/v1beta2
kind: LocalQueue
metadata:
  name: team-a-queue
spec:
  clusterQueue: team-a-cq

---
apiVersion: kueue.x-k8s.io/v1beta2
kind: LocalQueue
metadata:
  name: team-b-queue
spec:
  clusterQueue: team-b-cq
```

## 场景一：Cohort 内抢占（reclaim）

目标：team-b 借了 team-a 的配额，team-a 有新任务来时，把配额抢回来。

### Step 1：team-b 先借用 team-a 的配额

此时 team-a 没有任务，team-a 保底 4 核全部空闲。team-b 提交一个 5 核任务（2 核保底 + 3 核借用），能正常准入。

```yaml
---
apiVersion: batch/v1
kind: Job
metadata:
  name: job-b1
  labels:
    kueue.x-k8s.io/queue-name: team-b-queue
spec:
  template:
    spec:
      containers:
        - name: sleep
          image: docker.io/library/busybox:1.38.0
          command: ["sleep", "600"] # 睡 10 分钟，方便观察
          resources:
            requests:
              cpu: "5"
      restartPolicy: Never
```

确认 Job B1 已经运行：

```bash
kubectl get workloads
kubectl get pods -o wide
```

此时集群配额使用情况：

- team-a 使用 0 / 保底 4
- team-b 使用 5 / 保底 2（借用了 3 核）

### Step 2：team-a 提交任务，触发抢占（reclaim）

team-a 提交一个 3 核任务，属于自己的保底范围。但当前 team-a 的保底已经被 team-b 借走 3 核，只剩 1 核可用，不够 3 核 → 触发 reclaimWithinCohort 抢占。

```yaml
---
apiVersion: batch/v1
kind: Job
metadata:
  name: job-a1
  labels:
    kueue.x-k8s.io/queue-name: team-a-queue
spec:
  template:
    spec:
      containers:
        - name: sleep
          image: docker.io/library/busybox:1.38.0
          command: ["sleep", "60"]
          resources:
            requests:
              cpu: "3"
      restartPolicy: Never
```

### 观察抢占过程

```bash
# 看 workload 事件，会看到 "Preempted" 事件
kubectl describe workload | grep -A 3 -i preempt

# job-b1 对应的 workload 会被标记为不再 Admitted，Pod 被驱逐
kubectl get workloads
kubectl get pods
```

现象：

1. Kueue 发现 team-a 的 job-a1 在自己保底范围内但没配额，从 Cohort 内借用出去最多的 team-b 挑一个 Workload 抢占。
2. job-b1 的 Workload 被 evict，Pod 被删除，Workload 重新进入 Pending。
3. job-a1 得到 3 核配额，开始运行。
4. 等 job-a1 跑完（60 秒），配额释放，job-b1 会重新准入（如果 team-b 此时又能凑齐 5 核）。

## 场景二：同队列内基于优先级抢占

目标：同一个 team-b 内，高优先级任务能挤掉低优先级任务。

先清理场景一的资源：

```bash
kubectl delete job job-a1 job-b1 --ignore-not-found
```

### Step 1：提交一个低优先级任务，占满 team-b 的配额

```yaml
---
apiVersion: batch/v1
kind: Job
metadata:
  name: job-b-low
  labels:
    kueue.x-k8s.io/queue-name: team-b-queue
    kueue.x-k8s.io/priority-class: low-priority
spec:
  template:
    spec:
      containers:
        - name: sleep
          image: docker.io/library/busybox:1.38.0
          command: ["sleep", "600"]
          resources:
            requests:
              cpu: "2"
      restartPolicy: Never
```

现在 team-b 保底 2 核已经用完（且 team-a 也可能没空闲配额可借，或者你可以先把 team-a 也占满以确保只能通过抢占来腾配额）。

### Step 2：提交高优先级任务，触发同队列抢占

```yaml
---
apiVersion: batch/v1
kind: Job
metadata:
  name: job-b-high
  labels:
    kueue.x-k8s.io/queue-name: team-b-queue
    kueue.x-k8s.io/priority-class: high-priority
spec:
  template:
    spec:
      containers:
        - name: sleep
          image: docker.io/library/busybox:1.38.0
          command: ["sleep", "60"]
          resources:
            requests:
              cpu: "2"
      restartPolicy: Never
```

### 观察

```bash
kubectl get workloads -o wide
kubectl describe workload | grep -A 3 -i preempt
```

现象：

1. `job-b-high` 需要 2 核，team-b 已被 `job-b-low` 占满。
2. Kueue 根据 `withinClusterQueue: LowerPriority` 策略，发现 `job-b-low` 优先级更低，抢占它。
3. `job-b-low` 被 evict，`job-b-high` 得到配额运行。
4. `job-b-high` 跑完后，`job-b-low` 会重新准入。

## 抢占策略速查

`spec.preemption` 常用字段：

| 字段 | 取值 | 含义 |
|---|---|---|
| `reclaimWithinCohort` | `Never` | 不能跨队列抢占（默认） |
| - | `LowerPriority` | 只抢 Cohort 内借用配额且优先级更低的 Workload |
| - | `Any` | 抢 Cohort 内任何借用配额的 Workload |
| `withinClusterQueue` | `Never` | 队列内不抢占（默认） |
| - | `LowerPriority` | 队列内高优先级抢低优先级 |
| - | `LowerOrNewerEqualPriority` | 加上「同优先级但更新的」也可以被抢 |
| `borrowWithinCohort.policy` | `Never` / `LowerPriority` | 是否允许在抢占的同时继续借用配额 |

## 排查小贴士

- `kubectl get workloads` 看 `ADMITTED` 状态和 `AGE`。
- `kubectl describe workload <name>` 里 Events 会明确写出 `Preempted by workload ...` 或 `Evicted due to Preempted`。
- 如果没触发抢占，检查：
  - ClusterQueue 是否配置了 `preemption` 字段。
  - 优先级是否真的低（`kubectl get workload <name> -o yaml | grep priority`）。
  - 目标 Workload 是否处于「借用」状态（reclaim 只对借用的 Workload 生效）。
